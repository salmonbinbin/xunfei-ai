## 二、TTS语音合成功能修复

### 2.1 问题现象

用户点击AI消息的播放按钮时，控制台显示：
```
POST /api/chat/tts?text=...&voice=x4_lingxiaoyao_em 502 (Bad Gateway)
[TTS] Error: code=11200, message=licc failed
```

### 2.2 问题根因

**错误码 `11200` = `licc failed`** - 发音人(vcn)未经授权或不存在。

前端代码一直发送发音人参数 `x4_lingxiaoyao_em`，但这个发音人：
1. 未在讯飞控制台开通
2. 不在应用的可调用发音人列表中

后端默认发音人是有效的 `xiaoyan`（讯飞少女音），但前端传的值覆盖了默认值。

### 2.3 技术解决方案

修改前端两处文件，将发音人从无效的 `x4_lingxiaoyao_em` 改为有效的 `xiaoyan`：

**文件1：`frontend/src/api/chat.js`**
```javascript
// 修改前
export function textToSpeech(text, voice = 'x4_lingxiaoyao_em') {

// 修改后
export function textToSpeech(text, voice = 'xiaoyan') {
```

**文件2：`frontend/src/stores/chat.js`**
```javascript
// 修改前
async function speakText(text, voice = 'x4_lingxiaoyao_em') {

// 修改后
async function speakText(text, voice = 'xiaoyan') {
```

## 三、图片理解日志埋点开发

### 3.1 问题背景

用户反馈：已正常使用课表图片OCR识别功能，但管理端API统计中没有OCR记录。

### 3.2 问题调查

通过分析后端日志发现：

1. 用户使用的是 `/timetable/import/preview` 接口（日志第87行）
2. 该接口**不经过 OCR 服务** (`ocr_service.recognize_document`)
3. 而是直接使用**星火大模型 (XingHuo) 的图像理解功能** `xinghuo_service.image_understanding`

```python
# 日志证据
[xfyun] [XingHuo] image_understanding called, image size: 389354 bytes
```

### 3.3 两条不同的代码路径

| 接口 | 使用的服务 | 是否记录日志 |
|-----|----------|-------------|
| `/timetable/import/preview` | `xinghuo_service.image_understanding` | ❌ 日志缺失 |
| `/timetable/upload` | `ocr_service.recognize_document` | ✅ 已有日志 |

### 3.4 技术解决方案

在 `xinghuo_service.py` 的 `image_understanding` 方法中添加日志记录：

**文件：`backend/app/services/xinghuo_service.py`**

在方法成功返回前（第302-303行之后）添加：
```python
self.logger.info(f"[XingHuo] Image understanding result length: {len(result_text)}")
self.logger.info(f"[XingHuo] Raw result: {result_text[:2000]}")

# 记录图片理解API调用日志（异步，不阻塞）
try:
    asyncio.create_task(save_api_log(
        api_name="image_understanding",
        call_type="success",
        response_time_ms=0,
        user_type="student"
    ))
except Exception as log_err:
    self.logger.warning(f"[XingHuo] Failed to save API log: {log_err}")

# 解析JSON结果
```

## 四、涉及的技术点

### 4.1 讯飞开放平台API

| API名称 | 用途 | 日志api_name |
|-------|------|-------------|
| ASR (语音识别) | 录音转文字 | `asr` |
| TTS (语音合成) | 文字转语音 | `tts` |
| OCR (文字识别) | 图片文字识别 | `ocr` |
| Spark (星火大模型) | AI对话、文本处理 | `spark` |
| NLP (语义分析) | 情感分析 | `nlp` |
| Image Understanding (图片理解) | 课表图片AI解析 | `image_understanding` |

### 4.2 核心代码模式

**异步非阻塞日志记录**（所有服务中统一使用）：
```python
try:
    asyncio.create_task(save_api_log(
        api_name="xxx",
        call_type="success/fail",
        response_time_ms=xxx,
        user_type="student"
    ))
except Exception as log_err:
    self.logger.warning(f"[Service] Failed to save API log: {log_err}")
```

### 4.3 使用的技术栈

- **后端**：FastAPI + SQLAlchemy 2.0 (async) + aiomysql
- **前端**：Vue.js 3 + Pinia + Axios
- **数据库**：MySQL
- **日志服务**：`asyncio.create_task` 实现非阻塞异步写入

## 五、经验总结

### 5.1 TTS发音人问题

**教训**：前端和后端必须使用统一且已开通的发音人列表

**建议**：
- 发音人列表应在后端配置，前端只传递有效的枚举值
- 或后端对前端传入的发音人做校验，拒绝未授权的发音人

### 5.2 日志埋点覆盖问题

**发现**：不同功能可能使用不同的API实现路径

**例子**：
- 课表识别有两条路：OCR服务 vs 星火图片理解
- 需要确保所有实际被调用的API路径都有日志埋点

### 5.3 异步任务丢失风险

**发现**：`asyncio.create_task` 在FastAPI的请求上下文中可能被提前取消

**影响**：日志可能无法100%记录，但在高并发下这是可接受的 tradeoff
