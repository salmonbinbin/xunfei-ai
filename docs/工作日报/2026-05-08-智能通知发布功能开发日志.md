## 一、项目概述

### 1.1 项目背景

AI小商教师端开发进度：
- ✅ 教师登录注册
- ✅ 智能成绩管理
- 🔄 **智能通知发布**（本次开发）
- ⏳ 智能备课教案

本次任务是开发教师端的**智能通知发布**功能，参考学生端"校园活动助手-宣传文案生成"的模式，通过讯飞星火大模型辅助教师生成规范的通知文档，并支持一键发送到飞书群。

### 1.2 功能需求

| 功能 | 说明 |
|-----|------|
| 通知类型选择 | 6种类型：考试/会议/活动/放假/征稿/其他 |
| AI生成通知 | 输入主题+补充信息，AI生成规范通知 |
| 结果可编辑 | 生成后可手动修改内容 |
| 复制通知 | 一键复制到剪贴板 |
| 飞书推送 | 选择群组发送到飞书群 |

### 1.3 技术栈

| 层级 | 技术 | 说明 |
|-----|------|------|
| 前端框架 | Vue.js 3 + Vite | 组件化开发，参考TeacherGrade.vue |
| 前端UI | Element Plus | el-card, el-input, el-select, el-dialog |
| 后端框架 | FastAPI | 异步API框架 |
| 数据库 | MySQL 8.0 + SQLAlchemy | 异步ORM |
| AI能力 | 讯飞星火大模型 | 生成通知文案 |
| 飞书推送 | Feishu Webhook | 发送到飞书群 |

### 1.4 讯飞技术应用

本次开发使用的讯飞技术：

| 技术 | 用途 | 调用方式 |
|-----|------|---------|
| **星火大模型(Spark API)** | 生成通知文案 | HTTP API调用 `chat_with_retry` |

**星火大模型Prompt设计**：
```
你是一个广州商学院的行政秘书，负责生成格式规范的通知文档。

要求：
1. 标题格式：【通知】+ 主题
2. 开头称呼：尊敬的各位同学/老师
3. 正文分点列出，逻辑清晰，使用中文数字（一、二、三）
4. 结尾落款：广州商学院XX学院 + 日期
5. 语气正式，内容准确

输入信息：
- 通知类型：{type_desc}
- 通知主题：{topic}
- 补充信息：{additional_info}

请严格按照上述格式生成通知，直接输出通知内容，不要添加标题或说明文字。
```

## 三、问题排查与解决

### 3.1 问题一：生成失败（res.data.success为false）

**现象**：点击"AI生成通知"后显示"生成失败"

**排查过程**：
1. 添加后端详细日志
2. 添加前端console.log追踪
3. 发现后端返回格式问题

**根因**：后端返回的Schema没有`success`字段

**解决**：
1. 修改`NotificationGenerateResponse`添加`success=True`
2. 修改router手动构造返回格式：
```python
return {
    "success": True,
    "data": result
}
```

### 3.2 问题二：群组选择显示"no data"

**现象**：点击"发送到飞书群"后，群组下拉框显示"no data"

**排查过程**：
1. 检查`getGroups`API调用
2. 查看飞书服务返回数据

**根因**：
- `getGroups`是GET请求，直接返回数组`[{id, name, configured}, ...]`
- 但前端错误地检查了`res.data?.success`
- 飞书服务返回的格式是`{id, name, configured}`，不是`{group_id, name}`

**解决**：
1. 前端修复数据判断逻辑：
```javascript
if (Array.isArray(res.data)) {
  groups.value = res.data || []
} else if (res.data?.success) {
  groups.value = res.data.data || []
}
```

2. 后端service转换格式：
```python
return [
    {"group_id": g["id"], "name": g["name"]}
    for g in groups
]
```

### 3.3 问题三：结果不可编辑

**现象**：AI生成的通知只能看，不能修改

**解决**：
- 将只读的`<pre>{{ result }}</pre>`改为可编辑的`<el-input type="textarea">`
- 添加编辑样式：
```css
.result-textarea :deep(.el-textarea__inner) {
  font-family: 'Noto Sans SC', sans-serif;
  font-size: 14px;
  line-height: 1.8;
  background: #F8FAFC;
}
```

## 四、经验总结

### 4.1 讯飞API使用

| 经验 | 说明 |
|-----|------|
| 重试机制 | 使用`chat_with_retry`而不是`chat_completion`，自动3次重试 |
| Prompt设计 | 简洁明了，提供具体格式要求 |
| 温度参数 | 生成正式通知使用temperature=0.7，平衡准确性和创造性 |
| 错误处理 | 所有API调用必须try-catch，记录详细日志 |

### 4.2 前后端数据交互

| 经验 | 说明 |
|-----|------|
| 响应格式统一 | 约定`{success: true, data: {...}}`格式 |
| GET vs POST | GET直接返回数据数组，POST返回`{success, data, error}` |
| 嵌套层级 | Axios包装后数据在`response.data`，不是`response.data.data` |
| 调试日志 | 关键位置添加console.log，便于追踪问题 |

### 4.3 Vue组件开发

| 经验 | 说明 |
|-----|------|
| 参考现有代码 | 参考TeacherGrade.vue的实现模式 |
| 响应式状态 | 使用`ref`和`reactive`管理状态 |
| 加载状态 | `generating`控制按钮loading状态和结果区域显示 |
| 表单验证 | `canGenerate`计算属性控制按钮可点击状态 |

### 4.4 FastAPI开发

| 经验 | 说明 |
|-----|------|
| Schema定义 | 独立Schema文件，按模块划分 |
| 路由注册 | 在`__init__.py`统一导出，在`main.py`注册 |
| 异常处理 | 路由中try-catch，统一返回格式 |
| 日志记录 | 使用`logging.getLogger()`创建专用logger |
