## 一、用户状态字段迁移（is_active → status）

### 问题背景
原 User 模型使用 `is_active` 布尔字段标识用户状态，但产品需求要求支持**禁用原因**功能，需要更灵活的状态机制。

### 技术方案
将 `is_active` 迁移为 `status` 字符串字段：
- `active` - 正常状态
- `disabled` - 已禁用

新增 `disable_reason` 字段存储禁用原因。

### 实施内容

**1. 数据库迁移脚本** (`backend/scripts/update_users_status.sql`)
```sql
ALTER TABLE users ADD COLUMN status VARCHAR(20) DEFAULT 'active';
ALTER TABLE users ADD COLUMN disable_reason TEXT;
UPDATE users SET status = 'active' WHERE is_active = 1;
ALTER TABLE users DROP COLUMN is_active;
```

**2. User 模型修改** (`backend/app/models/user.py`)
```python
status = Column(String(20), default="active")  # active / disabled
disable_reason = Column(Text, nullable=True)  # 禁用原因
```

**3. 涉及文件修改**
- `backend/app/models/user.py` - 模型定义
- `backend/app/routers/auth.py` - 学生注册/登录
- `backend/app/routers/teacher/auth.py` - 教师注册/登录
- `backend/app/schemas/auth.py` - 响应模型
- `backend/app/utils/auth.py` - 认证依赖
- `backend/app/routers/admin_console/user.py` - 管理端用户管理

### 经验总结
数据库字段迁移需要确保向后兼容，先添加新字段再删除旧字段，避免数据丢失。

### 功能需求
1. 用户分页列表展示
2. 禁用/启用用户（需填写原因）
3. 被禁用用户登录时显示禁用原因

### 技术实现

**1. 学生/教师注册** - 写入 `status="active"`
```python
user = User(
    phone=register_data.phone,
    password_hash=password_hash,
    status="active",  # 新增
    role=UserRole.teacher
)
```

**2. 登录时禁用检查** (`backend/app/routers/auth.py`)
```python
if user.status == "disabled":
    disable_msg = "账号已被禁用"
    if user.disable_reason:
        disable_msg += f"，原因：{user.disable_reason}"
    raise HTTPException(status_code=403, detail=disable_msg)
```

**3. 管理端禁用操作** (`backend/app/routers/admin_console/user.py`)
```python
@router.post("/users/{user_id}/disable")
async def disable_user(user_id: int, req: DisableUserRequest, db: AsyncSession):
    # 更新 status 和 disable_reason
    user.status = "disabled"
    user.disable_reason = req.reason
```

## 三、注册500错误调试（系统性调试流程）

### Phase 1: 根因调查

**错误信息**
```
Failed to load resource: the server responded with a status of 500
```

**调试步骤**

1. **检查后端日志**
```bash
tail -100 backend.log | grep -A5 "register"
```

2. **追踪代码调用链**
- `app/routers/auth.py:register()` → 创建 User 对象 → `db.commit()`
- 错误发生在 User 对象创建阶段

3. **发现根因**
```python
# 原代码 - 错误
user = User(
    phone=register_data.username,
    password_hash=password_hash,
    nickname=register_data.nickname or f"用户{register_data.username[-4:]}",
    is_active=1  # ❌ User 模型已移除 is_active 字段
)
```

### Phase 2: 修复方案

| 文件 | 错误 | 修复 |
|------|------|------|
| `auth.py` | `is_active=1` | `status="active"` |
| `teacher/auth.py` | `is_active=1, role=UserRole.teacher` | `status="active", role=UserRole.teacher` |
| `utils/auth.py` | `if not user.is_active` | `if user.status == "disabled"` |
| `schemas/auth.py` | `is_active: bool` | `status: str` |

### Phase 3: 验证测试
```bash

curl -X POST http://localhost:8000/api/auth/register \
  -d '{"username":"13800138009","password":"test123","nickname":"测试用户"}'

  -d '{"phone":"13900000019","password":"test123","name":"测试教师","department":"测试系"}'
```

### 经验总结
**系统性调试**的重要性：
- 错误信息 500 是通用错误，需要深入日志追踪
- 数据库模型变更后，所有引用该模型的地方都需要检查

## 四、last_login 时区问题调试

### 问题现象
用户在实际下午 16:57 登录，但数据库记录的时间是早上 08:35，相差 8 小时。

### 根因分析

**代码中使用 `datetime.utcnow()`**
```python
# auth.py 第78行
user.last_login = datetime.utcnow()  # ❌ 返回 UTC 时间
```

**时区差异**
- Python `datetime.utcnow()`: UTC 时间（早上 08:xx）
- MySQL `NOW()`: 本地时间 CST（下午 16:xx）
- 时差：8 小时

### 修复方案
```python
# 修复前
user.last_login = datetime.utcnow()

# 修复后
user.last_login = datetime.now()  # 使用本地时间
```

**涉及文件**
- `backend/app/routers/auth.py`
- `backend/app/routers/teacher/auth.py`

### 验证结果
```bash
# 修复前 last_login: 2026-05-09 08:59:50
# 修复后 last_login: 2026-05-09 17:00:32  ✅ 与实际登录时间一致
```

### 经验总结
- **时区问题是隐蔽的时间 bomb**：代码在本地测试正常，但用户在不同地区使用时暴露
- 统一使用本地时间 `datetime.now()` 避免歧义
- 建议在数据库层面统一时区设置
