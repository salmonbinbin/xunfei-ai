## 一、教师端注册登录功能调试

### 1.1 问题描述
教师注册接口返回500错误，前端无法正常完成注册流程。

### 1.2 问题分析
通过查看后端日志发现错误：
```
(1054, "Unknown column 'is_deleted' in 'field list'")
[SQL: INSERT INTO teacher_profiles (user_id, department, office, title, is_deleted, created_at, updated_at) VALUES (...)]
```

**根本原因**：`teacher_profiles` 表缺少 `BaseModel` 要求的 `is_deleted` 软删除字段。在创建表时没有完整执行 BaseModel 的所有字段。

### 1.3 解决方案
执行数据库迁移添加缺失字段：
```sql
ALTER TABLE teacher_profiles ADD COLUMN is_deleted TINYINT(1) DEFAULT 0;
```

### 1.4 验证结果
- 教师注册：成功
- 教师登录：成功

### 1.5 经验总结
- 数据库表结构必须与 ORM 模型定义完全一致
- BaseModel 包含公共字段：`id`, `is_deleted`, `created_at`, `updated_at`
- 后续创建表时需要确保包含所有 BaseModel 字段

## 二、路由跳转问题修复

### 2.1 问题描述
教师注册成功后跳转到 `/guide`（学生端引导页）而不是 `/teacher`（教师端首页）。

### 2.2 问题分析
1. **hasProfile 计算逻辑问题**：
   - teacher 注册后 `userInfo` 被设置为 teacher 对象 `{id, phone, name, department, title}`
   - `hasProfile` 检查的是 `profile.major` 和 `profile.grade`（学生字段）
   - teacher 没有这个嵌套结构，所以 `hasProfile = false`

2. **role 判断不可靠**：
   - 之前用 `userInfo?.role === 'teacher'` 检查，但 teacher 的 userInfo 里没有 role 字段
   - 用 `token?.includes('teacher')` 也不行（JWT 字符串本身不含 'teacher'）

### 2.3 解决方案
从 JWT token 解码 payload 获取 role：
```javascript
const isTeacher = (() => {
  try {
    const token = userStore.token
    if (!token) return false
    const payload = JSON.parse(atob(token.split('.')[1]))
    return payload.role === 'teacher'
  } catch {
    return false
  }
})()
```

## 六、经验总结

### 6.1 数据库设计
- ORM 模型与数据库表结构必须保持一致
- BaseModel 的公共字段（is_deleted, created_at, updated_at）必须添加到所有表
- 建议：创建表时使用 ORM 的 create_all 或 alembic 迁移，避免手动 SQL

### 6.2 前端架构
- 公共布局组件（Layout）应该统一，避免重复代码
- 路由守卫中的 role 检查应该从 JWT token 解码获取，而不是依赖 userInfo
- 移动端适配需要考虑 safe-area-inset-bottom

### 6.3 调试技巧
- 后端日志中添加详细的 try-catch 和步骤记录
- 前端使用 console.error 打印完整 error 对象
- API 测试使用 curl 命令快速验证
